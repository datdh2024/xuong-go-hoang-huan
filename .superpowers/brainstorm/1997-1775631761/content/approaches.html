Dark Mode — Implementation Approaches

Three ways to implement dark mode with next-themes + Tailwind CSS v4

A

🏆 Recommended: Tailwind darkMode: 'class' + next-themes

How: Configure Tailwind's darkMode: 'class'. Wrap app in <ThemeProvider> from next-themes. Add dark: variant classes to every component.

Pros

  • Industry standard pattern — well documented
  • next-themes handles SSR, flash prevention, localStorage, system preference
  • Each component explicitly declares its dark styles
  • Easy to audit — grep for dark: to find all dark mode styles

Cons

  • Must touch every component to add dark: classes
  • HTML class strings get longer
B

CSS Variables Override

How: Define CSS variables for all colors in globals.css, then override them under .dark selector. Components use variables instead of direct color classes.

Pros

  • Components don't need dark: classes — colors switch automatically
  • Fewer changes per component

Cons

  • Requires rewriting ALL existing color classes to use CSS variables
  • Loses Tailwind's utility-first approach — harder to reason about colors
  • Massive refactor of existing components
C

Hybrid: CSS Variables for Base + dark: for Exceptions

How: Use CSS variables for the most common colors (bg, text) that auto-switch, but use dark: classes for component-specific overrides.

Pros

  • Fewer dark: classes needed per component
  • Base colors switch automatically

Cons

  • Two mental models — devs must know which approach applies where
  • Still requires CSS variable refactor for base colors
  • Harder to maintain consistency

💡 Recommendation: Approach A

The codebase already uses Tailwind utility classes everywhere. Adding dark: variants is the natural extension — no refactoring needed, just augmenting existing classes. next-themes handles all the tricky SSR/hydration/localStorage logic. This is the standard approach for Next.js + Tailwind projects.
